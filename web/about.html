Overview

From Prompt to Reusable Training Signal

Scenario creation is usually slow, manual, and difficult to scale. This project treats it as an LLM-driven workflow where generation, evaluation, and refinement are connected from the start.

A fine-tuned model extracts structured scenario parameters from natural-language traffic requests. A base LLM handles open-ended geometry and XML reasoning. SUMO executes the resulting scenario, while the backend logs what happened, what was corrected, and what can be reused as future training data.

The goal is not only to generate scenarios, but to preserve correction signals in a form that can be evaluated, exported, and fed back into future fine-tuning.

Natural language in Traffic requests are translated into structured scenario parameters.
Runnable scenario out Generated parameters and geometry become executable SUMO artifacts.
Human feedback preserved Correction logs remain reusable instead of disappearing into chat history.
What I Built

A Workflow, Not a Single Prompt

Each layer has a clear role in generation, refinement, evaluation, and future dataset improvement.

FT

Fine-Tuned Extraction

A fine-tuned model converts traffic requests into structured parameters such as speed, volume, lanes, speed limit, sigma, tau, and block length.

XML

Base LLM Geometry / XML

A separate base model handles geometry edits, XML generation, and modification classification when structured extraction alone is not enough.

RUN

SUMO Scenario Execution

Structured parameters and network artifacts are turned into runnable SUMO scenarios with generated network, route, additional, and config files.

C/T

Correction vs Tuning

User feedback is split into trainable corrections and non-trainable tuning requests, so preference edits do not contaminate retraining data.

ADM

Evaluation and Reporting

An admin layer exposes correction history, field-level error patterns, geometry correction categories, and downloadable retraining exports.

OPS

Deployment Readiness

The project includes local execution, Docker support, CI/CD setup, and a Cloud Run deployment path for serving the workflow beyond local development.

Training Data Paths

Three Data Sources Feed the Same Improvement Loop

Fine-tuning in this project is not based on a single source. The workflow combines synthetic supervision, real-data-derived targets, and correction-derived retraining signals.

Synthetic

Prompt-Target Scenario Generation

Synthetic prompt/target pairs are built from road priors, time-of-day patterns, weather conditions, and traffic engineering heuristics.

Used to bootstrap structured parameter extraction before live correction data accumulates.
Real Data

Observed Seoul Traffic Targets

Real speed and volume files are aggregated by road and time period, then converted into fine-tuning targets for structured scenario fields.

Used to anchor the model to observed road behavior instead of synthetic priors alone.
Corrections

Human-in-the-Loop Retraining Signals

Session logs preserve before/after states, correction intent, field deltas, and trainability metadata so validated errors can flow back into future datasets.

Used to turn interactive usage into reusable fine-tuning data while keeping tuning requests out by default.
How the Workflow Runs

Generation and Refinement Are Part of the Same Loop

The workflow starts from natural-language input, produces a structured scenario, executes it in SUMO, and keeps the result auditable through validation and logging.

Generation Flow

A traffic request is converted into structured parameters, used to build or retrieve geometry, executed in SUMO, and validated as a complete scenario run.

flowchart TD A[User Natural-Language Request] --> B[Fine-Tuned Parameter Extraction] B --> C[Structured Scenario Parameters] C --> D{Usable real location?} D -->|Yes| E[OSM-Based Network Retrieval] D -->|No or Failed| F[Base LLM Geometry or XML Generation] E --> G[SUMO Network Build] F --> G C --> H[Demand / Route / Config Generation] G --> I[Runnable SUMO Artifacts] H --> I I --> J[Scenario Execution] J --> K[Validation and Statistics]

Refinement and Retraining Flow

After execution, users continue with either Correction or Tuning. Only correction paths become retraining exports; tuning remains available for analysis.

flowchart TD A[Generated Scenario] --> B[User Feedback] B --> C{Intent} C -->|Correction| D[Trainable Signal] C -->|Tuning| E[Analysis Only] D --> F{Target} E --> F F -->|Parameter| G[Parameter Update] F -->|Geometry| H[Geometry or XML Update] F -->|Mixed| I[Combined Update] G --> J[Rebuild and Rerun] H --> J I --> J J --> K[Session Logging] K --> L{Trainable} L -->|Correction| M[Correction Export] L -->|Tuning| N[Analysis Store] M --> O[Future Fine-Tuning Data] O --> P[Updated Extraction Stage]
Correction -> trainable Verified error feedback is preserved as future retraining signal.
Tuning -> analysis only Alternative variants remain logged, but stay out of the trainable dataset by default.
Correction export -> FT dataset The system closes the loop by turning validated corrections back into structured training data.
Why this split matters
Separating Correction from Tuning keeps preference edits out of the retraining path. That makes the exported dataset cleaner, easier to trust, and more useful for future fine-tuning.

Current Limits

What Is Strong Today, and What Still Needs Work

The project is strongest in workflow design, correction logging, and retraining structure. The current limits are mostly around robustness and evaluation breadth rather than missing pipeline pieces.

Geometry editing remains more fragile than parameter extraction, especially under more open-ended or mixed modification requests.
Some external dependencies such as OSM and public traffic APIs can fail, which affects runtime reliability more than workflow design.
The project is more mature in LLM workflow design and data logging than in frontend polish.
Evaluation is currently stronger for correction-driven analysis than for a large held-out benchmark suite.
