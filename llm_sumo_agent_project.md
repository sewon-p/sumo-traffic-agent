# LLM-Powered Traffic Simulation Agent
## 자연어 기반 교통 시뮬레이션 자동 생성 시스템

---

## 1. 프로젝트 목표

사용자가 자연어로 교통 시나리오를 요청하면, LLM 에이전트가 다양한 데이터소스를 탐색·조합·보정하여 완성된 SUMO 시뮬레이션 파일을 자동 생성하는 시스템.

```
입력:  "강남역 퇴근시간 시뮬레이션해줘"
출력:  실행 가능한 SUMO XML 파일 세트 (.net.xml, .rou.xml, .add.xml, .sumocfg)
```

---

## 2. 핵심 가치

- SUMO XML 작성은 전문 지식이 필요하고, 수동 세팅에 수일~수주 소요
- 교통공학자도 SUMO 세팅에 고전하는 상황에서, 자연어 한 문장으로 시뮬레이션 생성
- 여러 공공 데이터소스를 자동으로 탐색·조합하여 현실적인 파라미터 결정
- 데이터가 불완전할 때 유사 도로/시간대 추천 및 보정

## 3. 시스템 아키텍처

```
┌─────────────────────────────────────────────────┐
│  사용자 입력 (자연어)                              │
│  "강남역 퇴근시간 시뮬레이션해줘"                    │
└──────────────────────┬──────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────┐
│  LLM Agent (Orchestrator)                       │
│  - 요청 파싱: 위치, 시간대, 조건 추출              │
│  - 필요 파라미터 목록 결정                         │
│  - tool-calling으로 데이터소스 탐색                 │
└──────────────────────┬──────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────┐
│  데이터 수집 Tools                                │
│                                                  │
│  Tool 1: OSM 기하구조 추출                        │
│    → 도로망, 차로수, 제한속도, 교차로, 곡선         │
│                                                  │
│  Tool 2: TOPIS API 교통 데이터                    │
│    → 링크별 속도 (5분 단위)                        │
│    → 검지기 교통량 (가용 지점)                      │
│                                                  │
│  Tool 3: 공공데이터포털                            │
│    → VDS 교통량, DTG 데이터                        │
│    → 차종 구성 (승용/화물/버스 비율)                │
│                                                  │
│  Tool 4: 통계 데이터                              │
│    → 서울시 교통통계 (시간대별/도로유형별)            │
│    → 요일별 패턴, 계절 보정                        │
│                                                  │
│  Tool 5: 유사 도로 검색 (fallback)                 │
│    → 요청 도로 데이터 없을 시                       │
│    → 유사 조건 도로 추천 + 보정 계수 적용            │
└──────────────────────┬──────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────┐
│  파라미터 결정 (LLM reasoning)                    │
│                                                  │
│  기하구조: OSM 기반 네트워크                       │
│  교통 수요: 시간대별 OD flow                       │
│  차종 구성: vType (승용 85%, 트럭 10%, 버스 5%)    │
│  운전자 성향: CF 파라미터 (tau, accel, decel,      │
│              sigma, speedFactor)                  │
│  신호 제어: 주기, 현시, offset (가용시 실데이터,    │
│            불가시 actuated 자동 생성)              │
│  기상/특수조건: 우천시 속도 감소 등                  │
│                                                  │
│  ※ 데이터 불완전 시:                               │
│    - 유사 도로 데이터로 대체 + 보정                  │
│    - FD 관계로 누락 변수 추정                       │
│    - 불확실성 범위 고지                             │
└──────────────────────┬──────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────┐
│  SUMO XML 생성                                   │
│                                                  │
│  .net.xml  → netconvert --osm-files 기반 생성     │
│  .rou.xml  → flow, route, vType 정의              │
│  .add.xml  → 검지기, 신호, 버스정류장              │
│  .sumocfg  → 전체 설정 통합                       │
│  run.sh    → 실행 스크립트                         │
└──────────────────────┬──────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────┐
│  검증 & 품질 평가                                 │
│                                                  │
│  SUMO 실행 → 시뮬레이션 결과 추출                  │
│  시뮬레이션 속도/교통량 vs 실데이터 비교             │
│  오차 분석 → 프롬프트/파라미터 개선 사이클           │
└─────────────────────────────────────────────────┘
```

## 4. 사용 시나리오 예시

### 시나리오 A: 정상 요청

```
사용자: "강남대로 퇴근시간 18~19시 시뮬레이션"

에이전트:
  1. OSM에서 강남대로 도로망 추출 (3차로, 50km/h, 1.2km)
  2. TOPIS에서 18~19시 평균속도 조회 → 22km/h
  3. 검지기 교통량 조회 → 1850 veh/h
  4. 서울시 통계 차종 비율 → 승용 87%, 트럭 8%, 버스 5%
  5. 한국 도시부 CF 파라미터 적용 (sigma=0.7, tau=1.2)
  6. SUMO XML 세트 생성
  7. 검증: 시뮬레이션 속도 24km/h vs 실데이터 22km/h → 오차 9%
```

### 시나리오 B: 데이터 불완전

```
사용자: "방배동 내부 이면도로 퇴근시간 시뮬레이션"

에이전트:
  1. OSM에서 도로망 추출 (1차로, 30km/h)
  2. TOPIS 속도 조회 → 해당 링크 데이터 없음
  3. fallback: 유사 조건 검색
     → "서초구 1차로 30km/h 이면도로" 데이터 존재
     → 18~19시 평균속도 15km/h, 추정 교통량 400 veh/h
  4. 사용자에게 고지: "방배동 데이터 미보유. 서초구 유사도로 기반 추정.
     교통량 400±100 veh/h로 설정합니다."
  5. SUMO XML 생성
```

### 시나리오 C: 조건 지정

```
사용자: "올림픽대로 여의도구간 사고 발생시 시뮬레이션.
        3차로 중 1차로 차단, 30분간"

에이전트:
  1. OSM에서 올림픽대로 여의도구간 추출 (3차로, 80km/h)
  2. TOPIS에서 평상시 교통량 조회
  3. 사고 시나리오: 특정 구간 1차로 폐쇄 설정
     → .add.xml에 variableSpeedSign 또는 closing rerouter 추가
  4. 수요는 평상시 그대로 유지 (사고 전 상태)
  5. SUMO XML 생성 + 사고 이벤트 XML 포함
```

## 5. 기술 스택

```
LLM:        OpenAI GPT-4o API (또는 Claude API)
Agent:      Python + function calling / tool-use
데이터:     TOPIS API, 공공데이터포털 API, OSM Overpass API
시뮬레이션: SUMO (netconvert, duarouter, sumo-gui)
검증:       Python (traci, xml.etree)
버전관리:   Git
배포:       Docker (선택)
```

## 6. 구현 단계

### Phase 1: 기본 파이프라인 (MVP)
- [ ] OSM → SUMO 네트워크 변환 자동화 (netconvert wrapper)
- [ ] TOPIS API 연결 (속도, 교통량 조회)
- [ ] LLM 프롬프트 설계: 자연어 → 파라미터 추출
- [ ] 기본 SUMO XML 템플릿 생성기
- [ ] 단일 도로 구간 end-to-end 데모

### Phase 2: 에이전트 고도화
- [ ] tool-calling 구조: LLM이 필요한 도구를 스스로 선택
- [ ] 유사 도로 검색 로직 (도로 유형, 차로수, 제한속도 매칭)
- [ ] 차종 구성, CF 파라미터 자동 결정
- [ ] 데이터 불완전 시 fallback 로직
- [ ] 다중 도로 구간 / 네트워크 지원

### Phase 3: 검증 & 개선 사이클
- [ ] SUMO 실행 후 결과 자동 추출
- [ ] 실데이터 vs 시뮬레이션 비교 리포트 자동 생성
- [ ] 오차 기반 파라미터 자동 재조정 (LLM이 오차를 보고 수정)
- [ ] 프롬프트 개선 이력 관리

### Phase 4: (선택) 확장
- [ ] 사고/공사/이벤트 시나리오 지원
- [ ] 시계열 시나리오 (24시간 연속)
- [ ] 다중 시나리오 배치 생성
- [ ] 웹 UI (Streamlit/Gradio)

## 8. 졸업과제와의 연결

```
졸업과제:
  SUMO → 프로브 데이터 → ML → 밀도 추정
  (SUMO 시나리오를 수동으로 만듦)

이 프로젝트:
  자연어 → LLM → SUMO 시나리오 자동 생성
  (SUMO 시나리오 생성 자체를 자동화)

연결점:
  - SUMO XML 구조에 대한 깊은 이해 (졸업과제에서 축적)
  - 교통 파라미터 의미를 아는 상태에서 LLM을 설계
  - "교통공학 지식 + LLM 활용" = 도메인 전문 AI 에이전트
```
